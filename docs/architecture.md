# システムアーキテクチャ詳細

`nuage-agent` は、GitHub の Issue/PR をトリガーにして複数の LLM 自律エージェントを自律分散実行する自動開発パイプラインシステムである。本ドキュメントでは、システムの並列処理モデル、ワークスペースの隔離機構、およびコード品質の維持メカニズムについて解説する。

---

## 1. 並列処理・実行スケジュールモデル

エージェント実行による競合（ファイルの同時編集やポートの衝突）を防ぎつつ、スピーディに開発サイクルを回すため、2層の実行キュー（タスクプール）を採用している。

```
                    [GitHub Issues / PRs]
                             │
                             ▼ (ポーリング・フィルタリング)
                     PipelineCrawler
                             │
            ┌────────────────┴────────────────┐
            ▼ (ポートやビルド等に影響あり)  ▼ (API・レビュー等影響なし)
       ConflictPool                      NonConflictPool
    (最大並行実行数: 1)                (最大並行実行数: 3)
            │                                 │
            ▼                                 ▼
       [TaskRunner]                      [TaskRunner]
```

### 競合考慮プール (ConflictPool)

- **対象タスク**: 開発 (`dev`, `dev-pr`)、QA検証 (`qa`)
- **最大並行数**: 1
- **理由**: テストやビルドを実際に走らせるタスクは、共通のポート利用やローカルDBの書き換え、ビルドキャッシュの上書きなどでコンフリクトを起こす可能性が高いため、リポジトリ単位で必ずシーケンシャル（順次）に処理を行う。

### 競合不要プール (NonConflictPool)

- **対象タスク**: 仕様定義 (`spec`)、コードレビュー (`review-general`, `review-semantic`)、QA自動起票 (`qa-generator`)
- **最大並行数**: 3
- **理由**: これらのタスクは GitHub API 経由でのテキスト対話や静的解析・差分レビューが主であり、ローカルビルド環境への競合影響がない。そのため並行度を高め、重いテスト処理を待つことなく即座に実行する。

## 2. Git Worktree によるワークスペースの完全隔離

単一リポジトリへの並行処理を行う際、同一のディレクトリでブランチの切り替え (`git checkout`) やファイルの編集を同時に行うと、Gitのインデックスロックや未コミットの競合が発生する。

これを排除するため、タスクの起動時にベースリポジトリ (`workspaces/<repo>/base`) から一時的な作業用コピーを作成する **Git Worktree** 機構を採用している。

### ライフサイクル

1. **ワークスペース準備**:
   タスクごとに隔離されたディレクトリ `workspaces/<repo>/task-<number>-<agentId>` を作成。
   - **Issue（通常開発）の場合**: `git worktree add -b <branchName> <taskDir> origin/<defaultBranch>` により新規作業ブランチを切った worktree を作成する。
   - **PR（レビュー/QA）の場合**: `--detach` オプションで worktree を作成したのち、`gh pr checkout` でPRの最新コミットをチェックアウトする。
2. **実行**:
   エージェント CLI は、この隔離された worktree ディレクトリを作業ディレクトリ (`cwd`) として実行される。生ログは worktree 外部の `workspaces/<repo>/logs/` ディレクトリに書き出され、Git トラッキングを汚さない。
3. **クリーンアップ**:
   タスクの終了（成功・失敗問わず）またはシャットダウン時に、`git worktree remove --force <taskDir>` および `git worktree prune` を実行して残存リソースを完全に削除する。

## 3. Graceful Shutdown

デーモンモードで動作中、`SIGINT` (Ctrl+C) または `SIGTERM` シグナルを受信した際の安全な終了メカニズムである。

- **新規受付のブロック**:
  シグナルを検知すると即座にシャットダウンフラグ (`isShuttingDown`) を立て、キューにある未実行のタスクを破棄し、新しいタスクの投入も受け付けない。
- **実行中プロセスの保護 (Drain)**:
  現在実行中の `claude` や `agy` CLI プロセスは強制終了せず、**自然完了するまで待機**する。
- **後処理の保証**:
  タスクが完了すると、`finally` ブロックが走り、GitHub の `agent:running` ラベルの剥奪や、作成された一時 Git Worktree の削除が自動的に完了する。すべての稼働タスクが終了した時点で、プロセスはステータス `0` で安全に終了する。

---

## 4. コード品質と単一責任原則の自動強制

コード規模の拡大に伴う関数の責務肥大化を防ぎ、かつ `@what` や `@why` などの要約の正確性を維持するため、ESLint による複雑度とサイズの制限ルールを導入している。

### ESLint ルールと設計意図

詳細な設定および各ルールの設計意図については、[eslint.config.ts](../eslint.config.ts) のコード内コメントを参照のこと。

### モジュール構造

クローラー (`src/runner/crawler.ts`) はディスパッチに専念し、実際の実行詳細は `src/runner/task-runner.ts` へ委譲されている。共通の GitHub 操作は `src/github/` パッケージ（`client.ts`, `issue.ts`, `pr.ts`, `label.ts`）に分割・集約し、ビジネスロジックとデータアクセス層、インフラ制御層を疎結合化している。
