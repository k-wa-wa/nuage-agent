# システムアーキテクチャ詳細

`nuage-agent` は、GitHub の Issue/PR をトリガーにして複数の LLM 自律エージェントを自律分散実行する自動開発パイプラインシステムです。本ドキュメントでは、システムの並列処理モデル、ワークスペースの隔離機構、およびコード品質の維持メカニズムについて解説します。

---

## 1. 並列処理・実行スケジュールモデル

エージェント実行による競合（ファイルの同時編集やポートの衝突）を防ぎつつ、スピーディに開発サイクルを回すため、2層の実行キュー（タスクプール）を採用しています。

```
                    [GitHub Issues / PRs]
                             │
                             ▼ (ポーリング・フィルタリング)
                     PipelineCrawler
                             │
            ┌────────────────┴────────────────┐
            ▼ (ポートやビルド等に影響あり)  ▼ (API・レビュー等影響なし)
       ConflictPool                      NonConflictPool
    (最大並行実行数: 1)                (最大並行実行数: 3)
            │                                 │
            ▼                                 ▼
       [TaskRunner]                      [TaskRunner]
```

### 競合考慮プール (ConflictPool)

- **対象タスク**: 開発 (`dev`, `dev-pr`)、QA検証 (`qa`)
- **最大並行数**: 1
- **理由**: テストやビルドを実際に走らせるタスクは、共通のポート利用やローカルDBの書き換え、ビルドキャッシュの上書きなどでコンフリクトを起こす可能性が高いため、リポジトリ単位で必ずシーケンシャル（順次）に処理を行います。

### 競合不要プール (NonConflictPool)

- **対象タスク**: 仕様定義 (`spec`)、コードレビュー (`review-general`, `review-semantic`)、QA自動起票 (`qa-generator`)
- **最大並行数**: 3
- **理由**: これらのタスクは GitHub API 経由でのテキスト対話や静的解析・差分レビューが主であり、ローカルビルド環境への競合影響がありません。そのため並行度を高め、重いテスト処理を待つことなく即座に実行されます。

---

## 2. Git Worktree によるワークスペースの完全隔離

単一リポジトリへの並行処理を行う際、同一のディレクトリでブランチの切り替え (`git checkout`) やファイルの編集を同時に行うと、Gitのインデックスロックや未コミットの競合が発生します。

これを排除するため、タスクの起動時にベースリポジトリ (`workspaces/<repo>/base`) から一時的な作業用コピーを作成する **Git Worktree** 機構を採用しています。

### ライフサイクル

1. **ワークスペース準備**:
   タスクごとに隔離されたディレクトリ `workspaces/<repo>/task-<number>-<agentId>` を作成。
   - **Issue（通常開発）の場合**: `git worktree add -b <branchName> <taskDir> origin/<defaultBranch>` により新規作業ブランチを切った worktree を作成。
   - **PR（レビュー/QA）の場合**: `--detach` オプションで worktree を作成したのち、`gh pr checkout` でPRの最新コミットをチェックアウト。
2. **実行**:
   エージェント CLI は、この隔離された worktree ディレクトリを作業ディレクトリ (`cwd`) として実行されます。生ログは worktree 外部の `workspaces/<repo>/logs/` ディレクトリに書き出され、Git トラッキングを汚しません。
3. **クリーンアップ**:
   タスクの終了（成功・失敗問わず）またはシャットダウン時に、`git worktree remove --force <taskDir>` および `git worktree prune` を実行して残存リソースを完全に削除します。

## 3. Graceful Shutdown (優雅な終了制御)

デーモンモードで動作中、`SIGINT` (Ctrl+C) または `SIGTERM` シグナルを受信した際の安全な終了メカニズムです。

- **新規受付のブロック**:
  シグナルを検知すると即座にシャットダウンフラグ (`isShuttingDown`) を立て、キューにある未実行のタスクを破棄し、新しいタスクの投入も受け付けません。
- **実行中プロセスの保護 (Drain)**:
  現在実行中の `claude` や `agy` CLI プロセスは強制終了せず、**自然完了するまで待機**します。
- **後処理の保証**:
  タスクが完了すると、`finally` ブロックが走り、GitHub の `agent:running` ラベルの剥奪や、作成された一時 Git Worktree の削除が自動的に完了します。すべての稼働タスクが終了した時点で、プロセスはステータス `0` で安全に終了します。

## 4. コード品質と単一責任原則の自動強制

コード規模の拡大に伴い、個々の関数やモジュールの責務が肥大化するのを防ぐため、ESLint によってコードの複雑度とサイズに厳しい制限を課しています。

### ESLint ルールと目的

- **`max-lines-per-function` (最大 50 行)**:
  関数が長大化すると、JSDoc の `@what` や `@why` で記述された要約の正確性が失われます。50行を超える場合は、強制的に小さな役割のヘルパー関数に分割させます。
- **`complexity` (最大 10)**:
  循環複雑度（if文やループ、論理演算子によるコード内の分岐経路数）を10以下に制限。単一の関数が複数の条件判断を持ちすぎるのを防ぎます。
- **`max-depth` (最大 4 階層)**:
  ネストが深くなるのを防ぎ、処理フローをフラットに保ちます。
- **`max-params` (最大 4 個)**:
  引数パラメータ数を4個以下に制限。引数が増えすぎた場合は、オプションオブジェクト（TypeScript インターフェース）へのカプセル化を強制します。

### モジュール構造

クローラー (`src/runner/crawler.ts`) はディスパッチに専念し、実際の実行詳細は `src/runner/task-runner.ts` へ委譲されています。共通の GitHub 操作は `src/core/github-client.ts` に集約し、ビジネスロジックとデータアクセス層、インフラ制御層を明確に疎結合化しています。
